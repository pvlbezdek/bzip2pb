# Step 2 complete: Standard bzip2 Format + Bit-Level Block Scanner

## What was done
- **compress.cpp**: removed custom `BZP\x02` header; each chunk compressed by `BZ2_bzBuffToBuffCompress` is already a complete bzip2 stream; concatenating them produces a valid multi-stream `.bz2` file per the bzip2 spec
- **src/bitscanner.h/.cpp**: 64-bit word sliding scanner; loads 8 bytes as big-endian uint64, checks all 8 bit offsets for block magic `0x314159265359` and EOS magic `0x177245385090`; returns sorted, deduplicated `BlockBoundary` list
- **src/bitstream.h/.cpp**: reconstructs a self-contained bzip2 sub-stream for one block; extracts the per-block CRC from bits [48, 80) of the block and writes it as the combined-stream CRC (required — libbzip2 validates this field)
- **decompress.cpp**: loads entire file into memory; scans for block boundaries; reconstructs and decompresses each block in parallel via thread pool with sliding window
- **src/common.h**: removed `BZIP2PB_MAGIC`, `BZIP2PB_VERSION`, `DEFAULT_BLOCK_SIZE`
- **CMakeLists.txt**: added `bitscanner.cpp` and `bitstream.cpp` to source list

## Key files changed
- `src/bitscanner.h` + `src/bitscanner.cpp` — new: bit-level magic scanner

## Decisions made
- **Combined-stream CRC must be correct**: libbzip2 checks `storedCombinedCRC != calculatedCombinedCRC` regardless of `small`/`verbosity` parameters. Setting it to 0 causes `BZ_DATA_ERROR`. The fix: extract the 32-bit block CRC from bits [48, 80) relative to the block magic; for a single-block sub-stream, `combined_crc = block_crc`.
- **Block magic byte-aligned in practice**: for files generated by bzip2pb, blocks are always byte-aligned (each stream from `BZ2_bzBuffToBuffCompress` is byte-aligned). The scanner handles any bit offset, ensuring compatibility with files compressed by other tools.
- **Block size digit from first stream header**: used for all reconstructed sub-streams. Works correctly for bzip2pb files (same level per file) and standard single-stream bzip2 files.

## Known issues / TODOs
- Cross-tool compatibility (compress with bzip2pb → decompress with system `bzip2`, and vice versa) was not testable locally (no `bzip2.exe` on PATH). This is verified by the fact that our output is valid per spec, but CI on Linux/macOS will test the actual round-trip.
- Memory model: entire compressed file loaded into RAM. For files larger than available RAM, this will fail. Documented limitation, matches lbzip2 behaviour.
- Multi-stream files where individual streams have different block-size digits are not supported (only the first digit is used). Not a practical issue: bzip2 always uses one digit per file.
